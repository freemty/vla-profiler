Key Observation

Model → System: Bubbles Are Inevitable

Robotics foundation models are evolving into multi-component pipelines. Pipelines have scheduling bubbles.

SystemComponentsPipeline Bubble
π0.7High-Level Policy → World Model → VLA → Action Expert4 serial stages, subgoal image transfer
GEN-1"system-level components across inference and model harnessing"Inference-time techniques (undisclosed)
Rhoda DVAVideo Model → IDM → ExecutionLeapfrog Inference: overlap video gen ∥ action exec
LingBot-VAVAE → Video DiT → Action DiT → KV Cache UpdateE→V→A serial + compute_kv_cache per chunk
Thesis: As models become systems, the gap between "model latency" and "system throughput" widens. Profiling the pipeline — not just the model — is where the next 2-5x lives.

Next Steps

Three Directions

A Algorithmic Reduction

Observe DiT attention → discover sparsity → sparse attn / step caching / token pruning

Analogy: FastVideo VSA / AutoGaze (4-100x pruning, CVPR'26)

Entry point: existing attention heatmaps (exp01b/05a) + per-step timing (exp04a)

B Pipeline Bubble Mining

LingBot-VA / Rhoda closed-loop pipeline: E→V→A serial wait + KV cache update

Analogy: DistServe PD disagg / Rhoda Leapfrog (video gen ∥ action exec)

Entry point: exp04d server log timing → quantify bubble → overlap feasibility

C Computation Ablation

Fast-WAM skip video → 94.5%. Which computations can be removed?

Ablation: remove video / reduce action steps / remove cross-attn / reduce layers → degradation curve

Entry point: existing step sweep (exp04c, exp09a) → extend to layer/head ablation

Common premise: Profiling data shows where to cut. 10 models × E/C/A breakdown = the scope on our blade.
